## 블로킹 vs 논블로킹

### 블로킹 I/O (Spring MVC)

```
요청 → 스레드 할당
스레드: DB 쿼리 요청 → [대기 중...] → 결과 수신 → 응답
```

스레드가 I/O 응답을 기다리는 동안 아무 일도 못 한다. 동시 요청이 늘면 스레드를 더 만들어야 한다. 스레드는 ~1MB 메모리를 소비한다.

**1000개 동시 요청** → 1000개 스레드 → ~1GB 메모리

### 논블로킹 I/O (Spring WebFlux)

```
요청 → 이벤트 루프 (소수의 스레드)
스레드: DB 쿼리 요청 → 콜백 등록 → 다른 요청 처리
...
DB 응답 → 콜백 실행 → 응답 반환
```

소수의 스레드(CPU 코어 수)가 이벤트 루프로 수천 개의 요청을 처리한다.

**1000개 동시 요청** → 8개 스레드 (CPU 코어 수)

## Reactor: Mono와 Flux

Spring WebFlux의 리액티브 타입.

```java
// Mono: 0 또는 1개의 데이터
Mono<User> userMono = Mono.just(new User(1L, "홍길동"));
Mono<User> empty = Mono.empty();
Mono<User> error = Mono.error(new UserNotFoundException(1L));

// Flux: 0개 이상의 데이터 스트림
Flux<User> usersFlux = Flux.just(user1, user2, user3);
Flux<Integer> numbers = Flux.range(1, 10);
```

### 주요 연산자

```java
// map: 동기 변환
Mono<String> name = userMono.map(User::getName);

// flatMap: 비동기 변환 (다른 Mono/Flux 반환)
Mono<Order> order = userMono.flatMap(user ->
    orderRepository.findLatestByUserId(user.getId())
);

// filter: 조건 필터
Flux<User> adults = usersFlux.filter(u -> u.getAge() >= 18);

// zipWith: 두 스트림 합치기
Mono<UserWithOrders> combined = userMono
    .zipWith(ordersMono)
    .map(tuple -> new UserWithOrders(tuple.getT1(), tuple.getT2()));
```

### 구독 (Subscribe)

리액티브 스트림은 **구독하기 전까지 실행되지 않는다** (lazy).

```java
// 직접 구독 (테스트/배치)
userMono.subscribe(
    user -> System.out.println("받은 데이터: " + user),
    error -> System.err.println("에러: " + error),
    () -> System.out.println("완료")
);

// WebFlux가 자동으로 구독 처리
@GetMapping("/users/{id}")
public Mono<User> getUser(@PathVariable Long id) {
    return userRepository.findById(id);  // subscribe는 WebFlux가 처리
}
```

## Spring WebFlux 구조

```
HTTP 요청
    ↓
Netty (비동기 서버)
    ↓
DispatcherHandler
    ↓
Handler Mapping → Handler Adapter → Controller
                                         ↓
                                    Mono/Flux 반환
                                         ↓
                                    Codec (직렬화)
                                         ↓
                                    HTTP 응답
```

Netty는 이벤트 루프 기반 비동기 서버다. Tomcat(스레드 기반)과 달리 소수 스레드로 수천 연결을 처리한다.

## WebFlux Controller

```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserRepository userRepository;
    private final OrderRepository orderRepository;

    // Mono 반환
    @GetMapping("/{id}")
    public Mono<ResponseEntity<UserDto>> getUser(@PathVariable Long id) {
        return userRepository.findById(id)
            .map(user -> ResponseEntity.ok(UserDto.from(user)))
            .defaultIfEmpty(ResponseEntity.notFound().build());
    }

    // Flux 반환 (스트리밍)
    @GetMapping(produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<UserDto> streamUsers() {
        return userRepository.findAll()
            .map(UserDto::from)
            .delayElements(Duration.ofMillis(100));  // SSE 스트리밍
    }

    // 병렬 처리
    @GetMapping("/{id}/dashboard")
    public Mono<UserDashboard> getDashboard(@PathVariable Long id) {
        Mono<User> user = userRepository.findById(id);
        Mono<List<Order>> orders = orderRepository.findByUserId(id).collectList();
        Mono<Long> totalAmount = orderRepository.sumAmountByUserId(id);

        return Mono.zip(user, orders, totalAmount)
            .map(tuple -> new UserDashboard(tuple.getT1(), tuple.getT2(), tuple.getT3()));
        // 3개가 병렬로 실행됨
    }
}
```

## R2DBC (리액티브 DB 드라이버)

WebFlux와 함께 사용하려면 DB 드라이버도 논블로킹이어야 한다.

```gradle
implementation 'org.springframework.boot:spring-boot-starter-data-r2dbc'
implementation 'org.postgresql:r2dbc-postgresql'
```

```java
public interface UserRepository extends ReactiveCrudRepository<User, Long> {
    Flux<User> findByStatus(UserStatus status);
    Mono<User> findByEmail(String email);
}
```

```java
// 트랜잭션
@Transactional
public Mono<Order> createOrder(CreateOrderRequest request) {
    return userRepository.findById(request.getUserId())
        .flatMap(user -> stockService.decreaseStock(request.getItems()))
        .flatMap(__ -> orderRepository.save(Order.create(request)));
}
```

## 에러 처리

```java
public Mono<UserDto> getUser(Long id) {
    return userRepository.findById(id)
        .switchIfEmpty(Mono.error(new UserNotFoundException(id)))
        .map(UserDto::from)
        .onErrorReturn(new UserDto()) // 에러 시 기본값
        .onErrorResume(UserNotFoundException.class, e ->
            Mono.just(UserDto.guest()) // 특정 예외만 처리
        )
        .onErrorMap(DataAccessException.class, e ->
            new ServiceException("DB 오류", e) // 예외 변환
        );
}
```

## WebFlux vs MVC 선택 기준

### WebFlux를 쓸 때

- 동시 연결 수가 매우 많은 경우 (채팅, 알림, IoT)
- I/O 집약적이고 CPU 작업이 적은 경우
- 마이크로서비스 간 많은 HTTP 호출이 있을 때
- SSE (Server-Sent Events), WebSocket

### MVC를 쓸 때

- 일반적인 CRUD 서비스
- 팀의 리액티브 학습 비용이 부담될 때
- JPA를 쓰는 경우 (JPA는 블로킹 → WebFlux의 이점 반감)
- 복잡한 비즈니스 로직 (리액티브 체인은 디버깅이 어려움)

### Java 21 Virtual Thread와의 비교

Java 21의 Virtual Thread는 블로킹 코드를 쓰면서도 높은 동시성을 달성한다.

```java
// Spring MVC + Virtual Thread (Java 21)
spring.threads.virtual.enabled: true

// 블로킹 코드 그대로 → JVM이 Virtual Thread로 처리
public User getUser(Long id) {
    return userRepository.findById(id); // 블로킹이지만 Virtual Thread가 OS 스레드 반납
}
```

WebFlux는 성능이 좋지만 학습 비용이 높다. Java 21 + Virtual Thread는 기존 코드 스타일을 유지하면서 비슷한 동시성을 달성할 수 있다.

## 면접에서 자주 나오는 질문

**Q. Spring MVC와 WebFlux의 핵심 차이는?**

MVC는 요청당 스레드를 할당하는 블로킹 모델이다. I/O 대기 시 스레드가 점유된다. WebFlux는 이벤트 루프 기반 논블로킹 모델이다. 소수의 스레드가 I/O 완료 이벤트를 처리해 더 많은 동시 요청을 처리할 수 있다.

**Q. Mono와 Flux의 차이는?**

Mono는 0 또는 1개의 데이터를 비동기적으로 나타낸다. 단건 조회, 저장에 사용. Flux는 0개 이상의 데이터 스트림이다. 목록 조회, 실시간 스트리밍에 사용.

**Q. 리액티브 스트림이 lazy하다는 게 무슨 의미인가?**

Mono나 Flux를 만들어도 subscribe가 호출되기 전까지 실제 실행이 일어나지 않는다. 조립(어떻게 처리할지 정의)과 실행(실제 데이터 흐름)이 분리된다. WebFlux는 HTTP 요청 처리 파이프라인 끝에서 자동으로 subscribe한다.

**Q. WebFlux를 쓸 때 JPA를 쓰면 안 되는 이유는?**

JPA는 블로킹 I/O를 사용한다. WebFlux의 이벤트 루프 스레드에서 블로킹 작업을 실행하면 해당 스레드가 멈춰 전체 이벤트 루프가 막힌다. 이를 피하려면 `Schedulers.boundedElastic()`으로 별도 스레드 풀에서 실행해야 하는데, 이러면 WebFlux의 이점이 줄어든다. R2DBC를 사용해야 완전한 논블로킹이 된다.
