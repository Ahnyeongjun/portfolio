## 문제: 서비스가 9개면 통신 규칙도 9배로 복잡해진다

모놀리식일 때는 "내부 통신"이라는 개념 자체가 없었다. 하나의 프로세스 안에서 함수를 호출하는 것뿐이었으니까. MSA로 쪼개고 나니 서비스 사이의 모든 호출이 네트워크를 타는 실제 통신이 됐고, 동시에 새로운 질문이 생겼다 - **어떤 서비스가 어떤 서비스와 통신할 수 있어야 하는가.**

기본 Kubernetes 네트워킹(kube-proxy 기반)은 이 질문에 답하지 않는다. 기본값은 "모든 파드가 모든 파드와 통신 가능"이다. 인증을 게이트웨이로 끌어올린 구조([이전 글](/blog/keycloak-envoy-gateway-auth) 참고)에서 "게이트웨이를 통과해야만 인증된 것"이라는 전제가 성립하려면, 백엔드 서비스를 게이트웨이 우회해서 직접 호출하는 경로 자체가 차단돼 있어야 한다. 이건 애플리케이션 코드가 아니라 네트워크 계층에서 강제해야 하는 문제다.

컨트롤플레인 쪽에는 또 다른 문제가 있었다. API 서버가 단일 노드에만 있으면 그 노드가 죽는 순간 클러스터 전체를 제어할 방법이 사라진다. 컨트롤플레인을 여러 노드로 이중화하더라도, 클라이언트(kubectl, kubelet, 다른 컨트롤플레인 컴포넌트)가 "어디로 접속해야 하는지"를 결정하는 고정된 진입점이 없으면 이중화 자체가 무의미해진다.

## 시도한 것

### 왜 하필 네트워크 계층인가 - kube-proxy가 안 하던 일

Kubernetes는 네트워킹을 직접 구현하지 않는다. **CNI(Container Network Interface)**라는 표준 인터페이스만 정의해 두고, 실제 파드 간 통신·정책 적용은 CNI 플러그인(Calico, Cilium 등)에 위임한다. kube-proxy는 이 CNI 위에서 "Service 이름 → 실제 파드 IP" 변환만 담당할 뿐, "이 파드가 저 파드를 호출해도 되는가"는 애초에 관여 범위 밖이다. 즉 지금까지 아무 문제가 없었던 게 아니라, **애초에 아무도 그 규칙을 강제하고 있지 않았던 것**이다.

<Accordion title="CNI가 없으면 파드 간 통신에 정책을 걸 수 없는 이유 (br_netfilter)">

리눅스 브리지(파드들을 연결하는 가상 스위치)를 통과하는 트래픽은 기본적으로 **iptables/넷필터를 거치지 않는다.** 브리지는 원래 "같은 스위치에 물린 장치끼리는 그냥 통과시킨다"가 기본 동작이기 때문이다.

`br_netfilter` 커널 모듈은 이 기본 동작을 뒤집어서, 브리지를 통과하는 트래픽도 iptables 체인을 거치게 만든다. 이 모듈이 로드되고 `net.bridge.bridge-nf-call-iptables = 1`이 설정돼 있어야만, kube-proxy의 Service NAT 규칙과 NetworkPolicy가 파드 간 통신에 실제로 적용된다.

정리하면:

```
br_netfilter 없음
  파드 브리지 트래픽 → 넷필터 우회 → iptables 규칙 무의미

br_netfilter 있음
  파드 브리지 트래픽 → iptables 체인 통과 → Service NAT · NetworkPolicy 적용 가능
```

CNI 플러그인은 이 커널 모듈 위에 정책 리소스(NetworkPolicy 등)를 얹어서, "누가 누구를 호출할 수 있는가"를 선언적으로 관리할 수 있게 해주는 계층이다. CNI가 없다면 이 규칙을 강제할 방법 자체가 없다.

또 하나 헷갈리기 쉬운 지점: CNI를 `kubectl apply`로 한 번 설치하면 클러스터 전체에 바로 적용된 것처럼 보이지만, 실제로는 **DaemonSet**(모든 노드에 파드 하나씩 자동 배치되는 워크로드)으로 정의돼 있어서 "노드마다 자동으로 복제돼 실행되는" 것이다. `kubectl create -f`는 API 서버에 리소스를 등록할 뿐이고, 실제로 각 노드에 에이전트를 심는 건 DaemonSet 컨트롤러가 한다. 노드가 새로 추가되면 그 노드에도 자동으로 CNI 에이전트 파드가 뜬다 - 사람이 SSH로 접속해서 뭔가를 설치할 필요가 없다.

</Accordion>

### Cilium - eBPF로 네트워크 정책을 한 계층에서

Cilium은 eBPF(커널 안에서 안전하게 실행되는 프로그램)를 기반으로 하는 CNI다. 기존 kube-proxy를 대체하면서, 네트워크 정책(L3/L4)뿐 아니라 L7(HTTP 경로·메서드 단위)까지 하나의 정책 계층에서 다룰 수 있다.

이걸 도입한 이유는 두 가지다.

1. **"게이트웨이만 백엔드를 호출할 수 있다"를 강제할 방법이 필요했다.** NetworkPolicy로 각 백엔드 서비스의 인그레스를 게이트웨이 네임스페이스/파드 셀렉터로 제한하면, 애플리케이션이 아무 것도 하지 않아도 우회 호출 자체가 네트워크 계층에서 막힌다. 인증을 게이트웨이로 옮기는 구조는 이 전제가 없으면 성립하지 않는다.
2. **가시성.** Cilium의 Hubble은 클러스터 내부의 실제 트래픽 흐름을 관찰할 수 있게 해준다. 어떤 서비스가 어떤 서비스에 접근을 시도했는지(허용됐든 거부됐든) 확인할 수 있어서, 예상 밖의 통신 시도가 있으면 - 예를 들어 있어서는 안 될 우회 호출 시도 - 정책을 만들기 전에도 눈으로 먼저 확인할 수 있다.

```
Before (kube-proxy, 정책 없음)
  모든 파드 ←→ 모든 파드 통신 가능
  "게이트웨이만 백엔드 호출 가능"은 코드 수준 신뢰에 의존

After (Cilium NetworkPolicy)
  백엔드 서비스 인그레스 = 게이트웨이 파드/네임스페이스만 허용
  우회 호출은 네트워크 계층에서 자체 차단
  Hubble로 실제 트래픽 흐름 관찰 가능
```

<Accordion title="iptables와 eBPF, 실제로 뭐가 다른가">

기존 방식(Calico 등 iptables 기반 CNI 다수 포함)에서 kube-proxy가 Service 라우팅을 처리하는 방법은 **iptables 규칙 체인**이다. 패킷 하나가 들어올 때마다 등록된 규칙을 위에서부터 순서대로 대조한다 - 규칙이 늘어날수록(서비스가 늘어날수록) 패킷 하나를 처리하는 데 확인해야 할 규칙 수가 함께 늘어난다. 서비스가 9개면 규칙도 그만큼 불어나는 구조다.

eBPF는 다르다. 커널 안에 작은 프로그램을 심어서, 패킷을 **해시맵 조회 한 번**으로 목적지를 찾는다. 규칙(서비스) 개수가 늘어나도 조회 비용이 거의 일정하다. Cilium이 kube-proxy를 완전히 대체(`kubeProxyReplacement=true`)하면, Service 라우팅 자체가 iptables 체인을 아예 거치지 않고 이 경로로 처리된다.

여기에 더해 iptables는 L3/L4(IP·포트) 단위까지만 본다. Cilium은 eBPF 프로그램이 애플리케이션 계층 프로토콜(HTTP)까지 파싱할 수 있어서, "이 파드는 호출 가능하지만 `DELETE /admin/*`은 안 된다" 같은 L7 정책도 같은 계층에서 표현할 수 있다.

**실제로 전환할 때 걸리는 부분**은 따로 있다. CNI를 완전히 갈아끼우는 작업은 무중단으로 하기엔 리스크가 크다 - 기존 파드들은 이전 CNI가 할당한 IP를 물고 있어서, IPAM(IP 주소 할당 방식) 자체가 바뀌면 모든 파드가 재생성돼야 한다. `kubeProxyReplacement=true`로 가려면 kubeadm이 기본으로 설치하는 kube-proxy DaemonSet도 함께 걷어내야 한다. 그래서 실제 전환 순서는 "각 노드 kubeadm reset → 컨트롤플레인 재초기화(`--skip-phases=addon/kube-proxy`로 kube-proxy 설치 자체를 건너뜀) → Cilium 설치 → 워커 재조인" 같은 **재구축에 가까운 절차**를 탄다.

</Accordion>

<Accordion title="프로덕션에 적용하기 전에 별도 랩에서 먼저 재현해봤다">

CNI를 통째로 바꾸는 작업이라 실 클러스터에서 바로 시도하기엔 실패 시 되돌리기가 어려웠다. 그래서 AWS EC2 위에 별도 학습용 클러스터를 구성해서, "기존 CNI 제거 → kube-proxy 제거 → Cilium 설치(`kubeProxyReplacement=true`) → 재조인" 절차를 먼저 통째로 리허설했다.

전환 후 정상 여부는 Cilium 자체 상태와 실제 트래픽 흐름 두 가지로 확인했다.

```
cilium status                # agent / operator / envoy 파드 전부 Ready
kubectl get ds -n kube-system kube-proxy   # NotFound → kube-proxy가 실제로 빠졌는지 확인
cilium status --brief        # KubeProxyReplacement: True (Direct Routing)
```

그리고 Hubble로 실제 트래픽을 관찰해서, 서비스 간 통신이 "우리가 의도한 경로로만" 흐르는지를 트래픽 레벨에서 직접 확인했다. 예를 들어 큐 기반으로 분리된 두 서비스가 메시지 브로커를 통해서만 간접적으로 연결돼 있어야 한다면, `hubble observe`로 캡처한 흐름에 두 서비스 사이의 **직접 연결이 단 한 번도 없다는 것** 자체가 아키텍처가 의도대로 동작한다는 증거가 된다. 정책 문서나 다이어그램이 아니라, 커널이 실제로 흘려보낸 패킷 기록으로 확인한 것이다.

이렇게 별도 환경에서 절차와 검증 방법을 먼저 굳혀두고 나서야, 실제 운영 클러스터에 같은 순서로 적용했다.

</Accordion>

### kube-vip - 컨트롤플레인에 고정 주소 하나

컨트롤플레인은 3개 노드로 이중화했다. 문제는 "3개 노드 중 어디에 접속해야 하는가"였다. 특정 노드의 IP를 그대로 API 서버 주소로 쓰면, 그 노드가 죽었을 때 접속 주소 자체가 사라진다.

kube-vip는 3개 컨트롤플레인 노드 앞에 **고정된 가상 IP(VIP)**를 두고, 실제로 어떤 노드가 이 VIP를 응답하고 있는지를 노드 상태에 따라 동적으로 옮긴다. 클라이언트(kubectl, kubelet, 다른 컴포넌트)는 항상 같은 VIP로만 접속하면 되고, 그 뒤에서 어떤 노드가 살아있는지는 kube-vip가 관리한다.

```
[ kubectl / kubelet / 클라이언트 ]
        │  항상 같은 VIP로 접속
        ▼
   [ kube-vip 관리 VIP ]
        │
   ┌────┼────┐
   ▼    ▼    ▼
 노드1  노드2  노드3   ← 컨트롤플레인 3개, 하나가 죽어도 VIP는 살아있는 노드로 이동
```

<Accordion title="VIP 페일오버가 실제로 동작하는 원리, 그리고 왜 이 방식은 클라우드에서 안 통하는가">

kube-vip의 ARP 모드는 원리가 단순하다. VIP를 들고 있는 노드가 "이 IP는 나한테 있다"는 **gratuitous ARP**(요청받지 않았는데도 스스로 보내는 ARP 응답)를 네트워크에 broadcast한다. 같은 L2 네트워크(같은 스위치 브로드캐스트 도메인)에 있는 모든 장치는 이 ARP를 받고 자신의 ARP 캐시를 갱신한다 - "VIP는 이제 이 MAC 주소로 가라"고. 노드가 죽으면 살아있는 다른 노드가 같은 VIP에 대해 gratuitous ARP를 다시 뿌려서 트래픽을 자기 쪽으로 가져온다.

이 방식이 성립하려면 전제가 하나 있다 - **VIP를 쓰는 모든 노드가 물리적으로 같은 L2 브로드캐스트 도메인에 있어야 한다.** 온프레미스·베어메탈 환경(이 클러스터가 kubeadm으로 베어메탈에 직접 부트스트랩한 이유이기도 하다)에서는 이게 당연히 성립한다.

반대로 AWS VPC 같은 클라우드 SDN(소프트웨어 정의 네트워크) 환경에서는 이 전제가 깨진다. VPC는 임의의 IP에 대한 ARP 브로드캐스트 자체를 허용하지 않는다 - 각 인스턴스는 자신의 ENI에 명시적으로 할당된 IP만 응답할 수 있고, "내가 이 IP를 새로 받았다"고 네트워크 전체에 방송하는 행위 자체가 VPC 패브릭에서 막혀 있다. 실제로 별도 AWS 랩 환경에서 L2/ARP 기반 VIP 방식(MetalLB의 L2 모드로 테스트)을 그대로 재현해봤을 때, ARP가 전파되지 않아 페일오버가 동작하지 않는 걸 직접 확인했다. 클라우드에서 같은 문제를 풀려면 BGP 기반 라우팅 광고나 클라우드 사업자의 로드밸런서 통합(AWS Load Balancer Controller 등)으로 접근해야 한다 - VIP를 "네트워크에 방송"하는 대신 "라우터에 등록"하는 방식으로 바꾸는 것이다.

정리하면, kube-vip의 ARP 모드는 마법이 아니라 "같은 스위치에 물린 장비는 ARP 방송을 신뢰한다"는 L2 네트워크의 기본 동작을 그대로 활용한 것이고, 그래서 이 클러스터가 베어메탈 환경이라는 전제와 떼어놓고 볼 수 없는 선택이다.

</Accordion>

## 결과

두 도구 모두 "장애가 안 나게 막는다"기보다 "장애가 나도 진입점과 통신 경계가 흔들리지 않게 한다"는 성격에 가깝다.

- Cilium 도입 이후, 게이트웨이를 우회한 백엔드 직접 호출은 애플리케이션 로직과 무관하게 네트워크 계층에서 차단된다. 인증 아키텍처([1편](/blog/keycloak-envoy-gateway-auth))의 "통과 = 인증됨" 전제가 실제로 성립하는 건 이 계층 덕분이다. kube-proxy를 걷어내고 eBPF 경로로 전환하면서 서비스가 늘어나도 규칙 조회 비용이 거의 그대로라는 점도 부수적으로 얻었다.
- kube-vip 덕분에 컨트롤플레인 노드 하나가 재시작되거나 장애가 나도, API 서버 접속 주소(VIP)는 그대로 유지된다. 클러스터를 조작하는 쪽에서는 노드 장애가 접속 문제로 이어지지 않는다. 다만 이 방식이 성립하는 건 베어메탈 L2 환경이라는 전제 위에서고, 같은 구성을 클라우드 VPC로 그대로 옮기면 동작 방식 자체를 바꿔야 한다는 것도 별도 랩 환경에서 미리 확인해 둔 부분이다.

두 구성 모두 평소에는 존재감이 없다. 진가는 노드 하나가 죽는 순간, 혹은 누군가 실수로 백엔드를 직접 호출하려는 순간에 드러난다.
