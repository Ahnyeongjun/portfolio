## RDBMS vs NoSQL

### RDBMS

- 스키마 고정 (테이블, 컬럼 미리 정의)
- SQL로 복잡한 조인과 집계
- ACID 트랜잭션 보장
- 수직 확장이 기본 (더 좋은 서버로)
- 예: PostgreSQL, MySQL, Oracle

### NoSQL

- 스키마 유연 (또는 없음)
- 조인 없음 또는 제한적
- BASE (Basically Available, Soft-state, Eventually consistent)
- 수평 확장 (서버를 추가)
- 예: MongoDB (문서), Redis (키-값), Cassandra (컬럼), Neo4j (그래프)

## CAP 이론

분산 시스템은 아래 3가지를 **동시에 모두 만족할 수 없다**.

**C (Consistency)**: 모든 노드가 같은 데이터를 본다.

**A (Availability)**: 요청은 항상 응답을 받는다 (성공 또는 실패).

**P (Partition Tolerance)**: 네트워크 분리가 발생해도 시스템이 동작한다.

실제 분산 시스템에서 네트워크 장애(Partition)는 항상 발생할 수 있다. 따라서 P는 포기할 수 없다.

결국 선택은 **CP** vs **AP**:

| 선택 | 특성 | 예시 |
|------|------|------|
| CP | 일관성 우선, 가용성 포기 | HBase, ZooKeeper, MongoDB(기본) |
| AP | 가용성 우선, 일관성 나중에 | Cassandra, CouchDB, DynamoDB |

**결제 시스템**: CP → 일관성이 중요. 잔액이 맞지 않으면 안 됨.  
**SNS 좋아요 수**: AP → 1~2개 차이는 허용. 항상 응답이 더 중요.

## MongoDB

**문서형 NoSQL**. JSON과 유사한 BSON 형식으로 저장.

```json
// users 컬렉션
{
  "_id": ObjectId("..."),
  "name": "홍길동",
  "email": "hong@test.com",
  "address": {                  // 중첩 문서
    "city": "서울",
    "zip": "12345"
  },
  "tags": ["backend", "java"],  // 배열
  "createdAt": ISODate("...")
}
```

### RDBMS와 다른 점

```
RDBMS           MongoDB
Table      →    Collection
Row        →    Document
Column     →    Field
JOIN       →    Embed 또는 $lookup
```

**Embedding (중첩)**: 관련 데이터를 하나의 문서에 포함. 조인 없이 한 번에 조회.

```json
// 주문에 상품 정보 포함
{
  "_id": "order-123",
  "userId": "user-789",
  "items": [
    { "productId": "p1", "name": "노트북", "price": 1500000 },
    { "productId": "p2", "name": "마우스", "price": 50000 }
  ],
  "total": 1550000
}
```

**Referencing (참조)**: RDBMS처럼 ID만 저장하고 `$lookup`으로 조인.

### 언제 MongoDB를 쓰는가

- 스키마가 자주 바뀌는 경우
- 계층적, 중첩 데이터가 많은 경우
- 빠른 개발 속도가 중요할 때
- 비정형 데이터 (로그, 이벤트)

### 주의사항

- 트랜잭션은 가능하지만 RDBMS보다 제한적이고 느림
- 조인이 필요한 복잡한 집계는 RDBMS가 더 적합
- 스키마 없다고 설계를 안 하면 결국 유지보수가 어려워짐

## Elasticsearch

**역색인(Inverted Index) 기반 검색 엔진**. 전문 검색(Full-Text Search)에 특화.

```
일반 인덱스 (B-Tree):
  행 → 컬럼값

역색인:
  단어 → 해당 단어가 포함된 문서 목록
  "노트북" → [doc1, doc5, doc9]
  "서울"   → [doc2, doc4, doc7]
```

단어로 즉시 문서를 찾을 수 있어 LIKE '%검색어%'보다 훨씬 빠르다.

### 주요 기능

```json
// 검색 쿼리
GET /products/_search
{
  "query": {
    "multi_match": {
      "query": "맥북 노트북",
      "fields": ["name^2", "description"],  // name에 가중치 2배
      "fuzziness": "AUTO"                    // 오타 허용
    }
  },
  "highlight": {
    "fields": { "name": {} }               // 검색어 하이라이팅
  }
}
```

**형태소 분석**: "노트북들이" → "노트북" (한국어 분석기 필요: nori)

### DB + Elasticsearch 조합

```
DB (PostgreSQL/MySQL) → 주요 데이터 저장, 트랜잭션
         ↓ 이벤트 기반 동기화
Elasticsearch → 검색 인덱스
```

DB가 소스 오브 트루스(Source of Truth). ES는 검색용 뷰.

## Cassandra

**와이드 컬럼 스토어**. 대규모 쓰기 집약적 워크로드에 특화.

**특성**:
- AP 선택: 가용성 > 일관성
- 수평 확장 우수 (노드 추가로 선형 성능 향상)
- 쓰기 성능이 매우 빠름 (LSM-Tree)
- 마스터 없음 (모든 노드가 동등)

**사용 케이스**: IoT 센서 데이터, 시계열 데이터, 로그 저장, 채팅 메시지 기록.

## 선택 가이드

| 요구사항 | 선택 |
|----------|------|
| 복잡한 관계, 트랜잭션 | PostgreSQL, MySQL |
| 빠른 단순 조회/캐시 | Redis |
| 전문 검색, 자동완성 | Elasticsearch |
| 계층적/유연한 스키마 | MongoDB |
| 대규모 쓰기, 시계열 | Cassandra |
| 그래프 관계 탐색 | Neo4j |

**실무 패턴**: 하나의 시스템에 여러 DB를 함께 쓴다.

```
주요 데이터: PostgreSQL (ACID, 복잡한 쿼리)
캐시:       Redis (빠른 조회)
검색:       Elasticsearch (전문 검색)
```

## 면접에서 자주 나오는 질문

**Q. NoSQL을 언제 선택하는가?**

스키마가 자주 변하거나 비정형 데이터를 다룰 때 MongoDB. 빠른 키-값 조회나 캐시는 Redis. 전문 검색은 Elasticsearch. 수평 확장이 중요하고 일관성보다 가용성이 우선인 대규모 쓰기 워크로드는 Cassandra.

**Q. CAP 이론에서 왜 세 가지를 동시에 만족할 수 없는가?**

네트워크 파티션(P)이 발생하면 노드 간 통신이 끊긴다. 이때 쓰기 요청이 오면 일부 노드에만 반영된다. 일관성(C)을 선택하면 다른 노드가 최신 데이터를 받을 때까지 요청을 거절해야 한다. 가용성(A)을 선택하면 구버전 데이터로 응답한다. 둘 다는 불가능하다.

**Q. Elasticsearch를 RDBMS 대신 쓸 수 없는 이유는?**

ES는 검색에 최적화되어 있지만 트랜잭션이 없고, 업데이트가 내부적으로 삭제+재삽입이라 느리다. 정확한 카운트 집계는 느리고, 복잡한 조인이 불가능하다. RDBMS가 소스 오브 트루스가 되고 ES는 검색 전용으로 사용하는 패턴이 일반적이다.

**Q. MongoDB의 단점은?**

기본적으로 스키마 검증이 없어 잘못된 데이터가 들어갈 수 있다. 트랜잭션 지원은 되지만 RDBMS보다 제한적이고 성능이 낮다. JOIN이 없어 여러 컬렉션 조합이 필요하면 애플리케이션 레벨에서 처리해야 한다. 메모리 사용량이 크다.
