## 1. 위성영상 플랫폼 MSA 전환

**회사:** 한컴인스페이스 | **기간:** 2022.12 ~ 현재  
**역할:** 아키텍처 재설계 주도 · 백엔드 엔지니어  
**스택:** `Spring Boot` `FastAPI` `RabbitMQ` `K8s` `Spring Cloud Gateway` `MyBatis`

### 왜 시작했나

모놀리식 서버에서 어드민 UI 버튼 하나를 수정해도 전체 서비스가 재시작됐습니다.  
재배포 건수 **월 10건**, 배포 한 번에 **4분 다운타임**.  
고객이 오류 문의를 하면 어느 컴포넌트 문제인지 파악까지 **1주일**이 걸렸습니다.

### 기술 선택 이유

**RabbitMQ를 선택한 이유:**  
Kafka는 처리량이 높지만 운영 오버헤드가 큽니다.  
우리 서비스는 위성 처리 작업 단위가 명확해서 메시지 큐 모델이 더 적합했고,  
ack/nack 기반의 확실한 전달 보장이 필요했습니다.

**Outbox 패턴을 직접 개발한 이유:**  
Debezium을 먼저 시도했으나 폐쇄망 환경에서 replication slot이 반복 파손되는 문제가 발생했습니다.  
CDC 인프라에 의존하지 않고 이벤트 유실을 막는 최소 구조가 필요했습니다.  
AOP + MyBatis 인터셉터로 비즈니스 트랜잭션과 동일한 트랜잭션 내에서 Outbox 테이블에 이벤트를 기록하는 경량 라이브러리를 직접 구현했습니다.

**Snowflake 알고리즘을 직접 구현한 이유:**  
폐쇄망과 외부망이 물리적으로 분리된 환경에서 양쪽이 동일한 이벤트 ID를 생성하면 충돌이 납니다.  
ZooKeeper 같은 외부 코디네이터를 사용할 수 없어 직접 구현했습니다.  
worker ID를 망별로 사전 할당하면 외부 조율 없이 충돌 없는 분산 ID를 생성할 수 있습니다.  
worker ID에 망 정보를 인코딩해서 파일 기반으로 전달된 로그에서 발생 서버도 즉시 추적 가능합니다.

## 2. Go 영상 타일링 API 고성능화

**기간:** 2024.04 ~ 2025.03  
**역할:** 단독 설계 및 개발  
**스택:** `Go` `GDAL` `Redis` `GeoTIFF` `Nginx` `K8s`

### 문제

지도 위에 위성 영상을 겹쳐 보여줄 때 WMS 방식을 사용하고 있었습니다.  
WMS는 요청마다 해당 바운더리의 영상을 동적으로 합성합니다.  
영상이 10장일 때는 괜찮았지만, 100장이 되자 API 응답이 **4초를 넘었습니다.**

### 왜 Java/Python이 아닌 Go였나

GeoTIFF → PNG 변환은 GDAL C 라이브러리를 직접 호출하는 CPU 집약 작업입니다.  
Go는 CGO로 GDAL을 직접 바인딩할 수 있고, 고루틴으로 타일 생성을 병렬화하기에 스레드 오버헤드가 낮습니다.  
Spring의 서블릿 스레드 모델보다 타일 생성 워크로드에 적합하다고 판단했습니다.

### 핵심 구현

**WMS → WMTS 전환:**  
타일 좌표(z/x/y)를 캐시 키로 고정하고 Go 고루틴으로 타일을 병렬 생성해 Redis에 적재합니다.  
동일 타일 재요청 시 합성 연산이 완전히 스킵됩니다.

**Nginx Ingress keepalive:**  
매 요청마다 TCP 핸드셰이크가 반복되던 오버헤드를 upstream keepalive로 제거해  
K8s Replica 확장성도 함께 확보했습니다.

### 결과

| 지표 | 이전 | 이후 |
|------|------|------|
| API 응답 속도 | 4초 이상 | 0.5초 미만 |

## 3. GPU 1장에서 70파드 병렬 추론

**기간:** 2024.10 ~ 현재  
**역할:** 인프라 설계 및 추론 서비스 개발  
**스택:** `Kubernetes` `Aliyun GPUShare` `ONNX Runtime` `FastAPI` `PyTorch`

### 문제

K8s 기본 GPU 스케줄러는 카운트 단위로 GPU를 할당합니다.  
추론 모델 1개당 GPU 전체를 독점하면 물리 GPU 1장에 파드 1개만 올라갑니다.  
실제 GPU 메모리 사용률은 10% 수준인데 자원의 90%가 유휴 상태였습니다.

### 기술 선택 이유

**Aliyun GPUShare를 선택한 이유:**  
NVIDIA MPS, MIG 등 다른 선택지도 검토했습니다.  
- MIG는 Ampere 이상 GPU에만 지원되는데 우리 환경 GPU는 해당하지 않았습니다.  
- MPS는 프로세스 수준 격리라 모델 로딩 실패 시 공유 컨텍스트 전체가 영향을 받을 수 있습니다.  
- GPUShare는 `aliyun.com/gpu-mem` 리소스 단위로 파드별 메모리를 선언해서  
  K8s 스케줄러가 직접 배치 결정을 내리는 구조라 운영이 단순했습니다.

**FastAPI 서비스를 3종으로 분리한 이유:**  
단일 서비스로 object-detection, segmentation, super-resolution을 모두 처리하면  
한 모델 업데이트 시 전체 서비스가 재시작됩니다.  
모델별로 FastAPI 서비스를 독립 배포해 각각 독립 스케일링이 가능하도록 설계했습니다.

### 결과

| 지표 | 이전 | 이후 |
|------|------|------|
| 파드 수 (GPU 1장) | 1개 | 70개 |
| GPU 활용률 | ~10% | 실 사용량 기준 운영 |
| 모델 업데이트 영향 | 전체 재시작 | 해당 서비스만 재시작 |

## 4. Mapin — 콘텐츠 관점 분석 AI 서비스

**기간:** 2026.03 ~ 04 | 사이드 프로젝트 (SWYP 앱 4기)  
**역할:** 백엔드 단독 설계 · 개발  
**스택:** `Spring Boot 4.0` `Java 21` `Virtual Threads` `GPT-4.1-mini` `PostgreSQL`  
**링크:** [App Store](https://apps.apple.com/kr/app/mapin/id6761400852) · [웹 대시보드](https://dashboard-phi-one-35.vercel.app/login)

### 왜 만들었나

유튜브를 볼수록 비슷한 관점의 영상만 계속 나오는 필터버블 문제를 겪었습니다.  
URL을 저장하면 AI가 콘텐츠 관점을 분석하고 반대 시각의 콘텐츠를 추천해주는 서비스입니다.

### 기술 선택 이유

**Virtual Threads (Spring Boot 4.0)를 선택한 이유:**  
URL 분석은 GPT API → YouTube API → Naver API를 호출하는 I/O 집약 작업입니다.  
전통적인 스레드 풀 모델에서는 동시 분석 요청이 많아지면 풀이 고갈됩니다.  
Virtual Thread는 I/O 블록 시 OS 스레드를 점유하지 않아 URL당 독립 스레드를 할당해도  
스레드 풀 한도 없이 처리할 수 있습니다.

**3단계 캐싱을 설계한 이유:**  
GPT API 비용이 가장 큰 변수였습니다. 캐싱 계층을 나눈 이유가 있습니다.

- **L1 (결과 JSON 캐시):** 동일 URL 재요청 시 GPT·검색 단계 전부 스킵
- **L2 (is_analyzed 플래그):** 소스 콘텐츠 분석 이력 재활용, GPT 분석 호출만 스킵
- **L3 (키워드 풀):** 키워드 기반 후보 DB 조회로 신규 추천에도 기존 분석 재활용

L2를 L1과 분리한 이유는 "재분석 없이 추천만 새로 받고 싶은" 케이스를 L2에서 잡아내기 위해서입니다.

**배치 스코어링을 선택한 이유:**  
반대관점 후보 N개를 개별 API 호출로 스코어링하면 N번 GPT를 호출합니다.  
후보 전체를 하나의 프롬프트에 넣어 1회 배치 처리하는 방식으로 토큰 비용을 줄였습니다.

### 테스트

E2E 포함 13개 테스트 파일 작성.  
`UserJourneyE2ETest`로 URL 저장 → 분석 → 추천 전 흐름을 단일 테스트로 검증했습니다.

### 결과

| 지표 | 결과 |
|------|------|
| 재요청 시 API 호출 | 0회 |
| 배포 | iOS App Store + 웹 대시보드 동시 배포 |

## 5. DeadlineMate — 스터디 모임 플랫폼

**기간:** 2026.03 ~ 04 | 팀 프로젝트 (FESI 13기)  
**역할:** 백엔드 (11개 도메인 설계)  
**스택:** `Spring Boot` `Java 21` `QueryDSL` `JUnit5` `GitHub Actions`  
**링크:** [GitHub](https://github.com/FESI13-3/FESI13-backend)

### 이 프로젝트에서 집중한 것

기능 구현보다 **"설계 결정을 어떻게 검증했는가"** 에 집중했습니다.

### 기술 선택 이유

**CQRS를 분리한 이유:**  
`GatheringService`(생성/수정/삭제)와 `GatheringQueryService`(목록/상세)를 분리했습니다.  
쿼리 최적화 작업이 커맨드 로직에 영향을 주지 않도록 하기 위해서입니다.  
실제로 목록 조회 N+1 문제를 IN 쿼리 일괄 처리로 교체할 때 커맨드 코드를 건드리지 않았습니다.

**`@TransactionalEventListener(AFTER_COMMIT)`를 선택한 이유:**  
알림 실패가 모임 생성 트랜잭션을 롤백시키면 안 됩니다.  
트랜잭션 커밋이 확정된 후에만 이벤트를 발행해서 외부 서비스 장애가 핵심 데이터에 영향을 주지 않도록 설계했습니다.

**PESSIMISTIC_WRITE를 선택한 이유:**  
동시 신청 시 maxMembers 초과 방지가 필요했습니다.  
낙관적 락은 충돌 시 재시도가 발생하는데 모임 신청처럼 사용자 경험에 직결되는 케이스에서는  
재시도보다 선점 락으로 즉시 차단하는 게 적합하다고 판단했습니다.

### 테스트 철학

51개 테스트 파일 — 단위·통합·E2E 3-layer 구조.

이벤트 발행이나 동시성 처리는 단위 테스트만으로는 검증이 불가합니다.  
`@SpringBootTest` E2E로 실제 트랜잭션 흐름까지 검증했습니다.  
`@Nested` + `@DisplayName` 한글 계층 구조로 테스트 의도를 명확히 했습니다.

### 결과

| 지표 | 결과 |
|------|------|
| 동시 신청 시 maxMembers 초과 | 0건 |
| 외부 서비스 장애 시 모임 데이터 유실 | 0건 |
| 테스트 자동화 | PR마다 GitHub Actions CI 자동 실행 |

## 6. 동시성 제어 콘서트 예약 시스템

**기간:** 2025.07 ~ 09 | 항해99 백엔드코스 9기 (상위 10% 수료)  
**역할:** 백엔드 (단독)  
**스택:** `Spring Boot` `Redis` `Kafka` `k6` `InfluxDB` `Grafana` `JUnit5`  
**링크:** [GitHub](https://github.com/hanghae99-backend/2-SERVICE)

### 이 프로젝트에서 집중한 것

동시성 제어 방식을 하나씩 직접 비교 검증했습니다.  
DB 비관적 락 → Redis 분산락(simple·spin·pub-sub) → 낙관적 락 순으로 전환하며  
각 방식의 트레이드오프를 측정했습니다.

### 핵심 의사결정

**낙관적 락으로 최종 전환한 이유:**  
분산락 방식에서 k6 + Grafana 부하 테스트를 돌려보니  
재시도 로직이 응답 지연의 주 원인임을 수치로 확인했습니다.  
좌석 예약은 충돌 빈도보다 처리 속도가 더 중요한 케이스여서 낙관적 락으로 전환했습니다.

가설을 세우고 → 측정으로 증명한 후 전환을 결정한 과정이 핵심입니다.

**Redis → Kafka로 마이그레이션한 이유:**  
Redis 대기열은 영속성이 없어 장애 시 메시지 유실 위험이 있습니다.  
예약·결제 이벤트를 별도 토픽으로 분리하고  
Choreography Saga + Redis DLQ로 분산 트랜잭션 실패 시 보상 이벤트를 처리했습니다.

### 결과

- 항해99 백엔드코스 9기 **상위 10% 수료**
- 동시성 제어 3가지 방식의 트레이드오프 직접 측정 및 검증
